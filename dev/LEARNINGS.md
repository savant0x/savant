## Session 2026-07-13-1900: v0.0.3 Release — Perfection Loop + Dev Guard Security + Tauri Version Alignment

**Key Learnings:**

- **LESSON-019 Perfection Loop pattern (43 issues closed in one release)** — the ECHO Law 13 utility-first + Law 12 redact() + Law 14 swallowed-error pattern: (a) identify a category of issues across the codebase (scattered `console.*` calls, `eslint-disable` comments, swallowed errors, missing test coverage); (b) build a single utility-first solution (`src/lib/logger.ts` with `info`/`warn`/`error` methods + `redact()` helper covering secret/password/token/credential/api_key/master_key/session_key/sub_key/bearer/authorization/hash/key$ patterns); (c) write comprehensive tests for the utility (17 cases covering secret redaction, case-insensitivity, non-mutation, primitives, nested objects, arrays of objects, nested arrays, non-plain objects like Date/Map/Set/RegExp, `Object.create(null)` cross-realm, plural key forms, key-secret-trumps-value-shape precedence); (d) propagate the utility to all call sites (6 `console.*` → `logger.*` replacements in dashboard-shell, use-derived-rotation, ipc, mock-ipc). The 43-issue loop went through 5 code-reviewer passes + 3 self-correction rounds to reach ECHO-compliant convergence. Cross-ref: CHANGELOG.md v0.0.3 `### Added (FID-009 — Perfection Loop)`.
- **LESSON-020 Tauri version alignment to renderer** — workspace `Cargo.toml` + `src-tauri/tauri.conf.json` were at `0.2.0` while the renderer (`VERSION`, `package.json`, `protocol.config.yaml`) was at `0.0.2`. The 0.2.0 was a premature bump from a prior drift session. Per the release-only-versioning discipline (existing `Versions rock ONLY at release time` entry above), all version files must mirror `package.json`. At the v0.0.3 release cut, Tauri was aligned from 0.2.0 → 0.0.3 to match the renderer. For a Tauri-first project, the Tauri app version and renderer version should ship together as a single versioned artifact.
- **LESSON-021 Dev guard security pattern for `output: "export"`** — with `output: "export"`, Next.js compiles route handlers to static JSON files at build time. Returning sensitive data (e.g., a master API key) in the response bakes it into the static bundle, which ships in the Tauri app. Anyone who extracts the bundle can see the key in plaintext. The fix: gate the full data on `NODE_ENV !== "production"` (dev only); in production, return a redacted summary (`last4` + `source`). The Tauri app uses Rust IPC for the env var in production, so the env var tier remains functional. The dev guard preserves dev convenience (env var tier 1 works without a Settings page visit) while securing the production build. Pattern is reusable for any route that returns sensitive data in a static export. A build-target flag (`NEXT_PUBLIC_BUILD_TARGET === "tauri"`) would NOT work — with `output: "export"`, the route is compiled to a static file and physically cannot read server env vars at runtime in a hypothetical non-Tauri web deployment. The Tauri app reads the env var server-side via Rust IPC in production, so the env var tier remains functional there. Cross-ref: `src/app/api/env/route.ts`, README.md §Web deployment.
- **LESSON-022 Markdownlint MD024 disable for changelogs** — Keep a Changelog format uses duplicate `## Added` / `## Fixed` headings across versions (every version has the same subsections). MD024 (no-duplicate-heading) flags this as an error. The fix: disable MD024 in `.markdownlint.json` at the project level (not per-file). Document the decision in the config file so future maintainers know it's intentional. Cross-ref: `.markdownlint.json`.

**Technical Insights:**

- **Next.js `force-static` vs `force-dynamic` with `output: "export"`** — `force-dynamic` is incompatible with static export (throws "cannot be used with output: export"). `force-static` is the correct choice for build-time-baked routes. Without any dynamic export, Next.js requires `force-static` or `revalidate` to be explicit. Cross-ref: `src/app/api/env/route.ts`.
- **`Object.getPrototypeOf(v) === Object.prototype` vs `v.constructor === Object`** — the `Object.getPrototypeOf` check correctly handles `Object.create(null)` (no prototype, no constructor) and cross-realm objects (e.g., from iframes). The `v.constructor` check fails for both cases. The double-cast `redact(v as unknown as LogContext)` acknowledges the type-system gap: TS can't statically prove that a value whose prototype matches `Object.prototype` is structurally a `LogContext`. The runtime is safe because `redact()` only reads `Object.entries(v)` and never mutates. Cross-ref: `src/lib/logger.ts`.
- **`grep` over-counts comment lines** — when checking `console.*` or `eslint-disable` counts, naive `grep` includes comment lines (e.g., `// console.log` or `// eslint-disable-next-line` in doc comments). The correct check excludes lines that start with `//` or `/*`. The 43-issue loop initially reported 6 `console.*` and 7 `eslint-disable` (including comments); the actual CODE counts were 3 and 4. Pattern: use `grep -v "// .*console"` to exclude comment lines.
