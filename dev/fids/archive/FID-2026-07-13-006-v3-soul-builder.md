## Detailed Description

### Problem

FID-006 v2 built a **read-only viewer** of an existing `workspace-savant/SOUL.md`. The actual feature in savant-orig is an **interactive authoring flow**:

- `dashboard/src/app/page.tsx:80-87` has a form with name input + prompt textarea + BootstrapTier buttons + a "Generate" button onClick that calls `sendControlFrame("SoulManifest", { prompt, name, bootstrap_tier })`.
- The Rust gateway receives the `ControlFrame::SoulManifest` frame and calls `generate_template_soul(prompt, name, birth_date)` at `crates/gateway/src/handlers/mod.rs:2031`, which produces an 18-section markdown template (the build-time default for the agent's SOUL.md).
- The rendered SOUL.md is written to disk by `load_agent` (`registry.rs:192-196` disk loader), and the chat's `Persona (SOUL)` formatted string picks up the new soul.

The v2 viewer had no input box. v3 builds the input box + dispatch + status panel + BulkManifest panel + Active Baseline display.

### Expected Behavior

After this FID:

- `src/types/control-frames.ts` (NEW) — TypeScript mirrors of `ControlFrame::SoulManifest { prompt, name, bootstrap_tier }`, `ControlFrame::BulkManifest { agents }`, `BootstrapTier` (4 variants), `BootstrapStatus` (3 variants), `AgentManifestPlan` (3 fields). All snake_case per `#[serde(rename_all = "snake_case")]` at `types/mod.rs:1154, 1174`.
- `src/lib/mock-ipc.ts` (EDIT) — extend with two new cases:
  - `case "manifest_soul":` — calls a `generate_template_soul(prompt, name, birth_date)` function (mirror of `handlers/mod.rs:2031`) and returns the templated string. Persists the result to a module-scoped `builtSouls` map.
  - `case "bulk_manifest":` — validates `agents.length <= 10` (SEC #8 limit per `handlers/mod.rs:647-655`), returns `{ status: "SWARM_DEPLOYED", count: agents.length }`.
- `src/lib/manifest-mock.ts` (NEW) — the `generate_template_soul` template-shape mirror (the 18-section markdown template, with a Phase-1 faithful subset of sections 1-3 + a marker for the rest). Exported as a pure function so the manifest page can call it directly for the form's status preview (Pending → Ready transition is UI-driven via `setTimeout` in the page).
- `src/app/manifest/page.tsx` (REWRITE) — 4 sections:
  1. **Soul Manifestation Engine** (builder form) — name text input, prompt textarea, BootstrapTier select (4 options), "Manifest Soul" submit button.
  2. **Draft Buffer** (persistence preview) — reads `localStorage["LS_SOUL_BUILT"]`, shows the most recent built soul with a BootstrapStatus badge (Ready/Pending/Degraded), a "Revert" button to clear the localStorage entry, and a "Copy to Clipboard" button.
  3. **Swarm Deployment** (BulkManifest) — JSON textarea for `AgentManifestPlan[]`, "Deploy Swarm" submit button, success metric display (`{ status, count }`).
  4. **Active Baseline** (v2 heritage) — the existing 3 cards (Identity & Vibe, Evolution State, Operating Directives) reading from `@/lib/soul`.
- The chat's `src/app/chat/page.tsx` is **unchanged** — its `import { SOUL_PROMPT } from "@/lib/soul"` continues to read the build-time canonical persona per LESSON-018.
- `npm run build` (output: `"export"`) succeeds; `?raw` webpack rule for `.md` files remains unchanged.

### Root Cause

FID-006 v2 anchored to the **consumer side** of the soul system (the disk-resident `SOUL.md` reader at `registry.rs:192` + the `ContextAssembler.assemble_system_prompt` section format) and built a viewer of the artifact. The **producer side** (the `ControlFrame::SoulManifest` dispatch + `generate_template_soul` template) was the missing piece. This FID closes the gap.

## Proposed Solution

## Lessons Learned

- **LESSON-016 reinforced (DRIFT-CLASS CLOSED).** FID-006 v2 anchored to the consumer side of the soul system; FID-006 v3 anchors to the **producer side**. The drift class that produced this — "paraphrase the feature name, then build a viewer of the resulting artifact" — is now closed. Future FIDs must trace the full control-flow: form → dispatch → handler → artifact → consumer, not just the consumer side.
- **LESSON-018 reinforced (BUILD-TIME INVARIANT).** The chat's `SOUL_PROMPT` continues to read the build-time canonical `workspace-savant/SOUL.md` via `?raw` import. The builder's output goes to `localStorage`, not to the build-time bundle. The build-time canonical is preserved; the builder is a separate authoring surface that persists across reloads but does not regress the chat's persona source.
- **PHASE 1 IPC PATTERN.** The `mockIPC` command-name pattern (string + payload dict) is the canonical Phase 1 IPC bridge. New IPC commands follow the `case "<command>":` shape in `src/lib/mock-ipc.ts`. The Tauri command names are strings, not TypeScript symbols, so this is a duck-typed contract — easy to mock, easy to swap to real Tauri in Phase 2.
- **EXACT NAMES, NOT UX RENAME.** The v2 outline renamed `BootstrapStatus::Success` → `Ready` (this was actually correct; the prior pasteback was wrong). The v3 pasteback confirms the exact names. v3 uses the exact savant-orig names with no UX rename. If a future FID needs a UX rename, do it presentationally in the JSX, not in the type.
- **SEC #8 IS A HARD LIMIT.** The bulk-manifest limit of 10 agents is enforced server-side at `handlers/mod.rs:647-655`. The renderer mock must mirror this — the `bulk_manifest` mock case returns `{ status: "REJECTED", reason: "SEC_8_LIMIT_EXCEEDED" }` for `agents.length > 10`. There is no soft limit; it's a constant.
- **THE 18-SECTION TEMPLATE IS A PHASE 1 SUBSET.** The Rust `generate_template_soul` produces an 18-section template (~200 lines). The renderer Phase 1 produces a faithful subset of sections 1-3 + a marker. The full template belongs in a future Rust-side mock or a real daemon call. The Phase 1 subset is a structural placeholder, not a feature degradation — chat's `SOUL_PROMPT` still reads the build-time `?raw` SOUL.md, which is the canonical baseline regardless of what the builder produces.
