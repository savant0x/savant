## Lessons Learned

(Captured in `dev/LEARNINGS.md` as LESSON-026 + LESSON-027.)

- **LESSON-026 str_replace multi-line phrase-split bug + SPACE-vs-HYPHEN-MINUS byte-level anti-pattern** ‚Äî the cascade-doc-consolidation exercise surfaced TWO related tool-fragility patterns. (1) SPACE-vs-HYPHEN-MINUS byte-level discrimination failures: when authoring an oldString for a multi-line block with a 5-7 word paragraph + punctuation separator, a single-byte hypothesis error at the prefix position causes the entire substring match to fail. Symptom: 8+ str_replace retries all report "not found" while the file's content is provably on disk. Resolution: Python `repr()` + per-byte hex+ascii dump; identify the mismatched byte (40-60% of the time it's space-vs-hyphen at a word-boundary). (2) Phrase-across-line-breaks substring-match invariant failure: when the canonical phrase is split across 2 lines, substring-mode `git grep -c` WILL NOT match unless normalized first. Symptom: forward-pointer text appears semantically correct, but `git grep -c` reports N-1 hits. Resolution: str_replace merging multi-line text into a single line. Both patterns cost 30+ minutes of debugging per incident; codified here to prevent future regressions.
- **LESSON-027 Doc-drift substring-match invariant design** ‚Äî with 1 canonical + N forward-pointers all using the EXACT canonical phrase as their anchor, `git grep -c '<canonical phrase>'` becomes the doc-drift linter's invariant. Drift becomes a substring-match problem instead of a content-equivalence problem. Companion tooling (`pnpm lint:docs`) is a recommended next step.

- **Pre-investigate before force-pushing** ‚Äî when a `git push origin <tag>` is rejected with "tag already exists", the right move is NOT blindly `--force`. First inspect the existing tag's content (`git cat-file -t <tag>` + `git tag -l <tag> -n99` + `git rev-parse <tag>`); verify whether the existing tag is complete vs. a prior pause-attempt; then use `--force-with-lease` to overwrite only after confirming the remote's current SHA matches expectations (the lease check is the safety net that catches concurrent overwrites).
- **`--skip-tag` flag for release.py** ‚Äî when the tag is already pushed (e.g., via force-with-lease), `release.py`'s normal flow would try to push the tag again (which would fail). The `--skip-tag` flag works around this ‚Äî proceeds directly to the GitHub Release API call without the tag step. The flag is documented in `scripts/release.py`'s `--help`.
- **Sub-agent clarification on the release.py pre-flight** ‚Äî `release.py`'s `[FAIL] Tag <v> already exists` check is local-only (it doesn't check the remote first). This is by design (the script's philosophy is "tag is local-first; push is a separate concern"). The consequence is that force-with-lease push + `--skip-tag release.py` is the canonical recovery pattern when the remote has a stale tag.
- **`force-with-lease` over plain `force`** ‚Äî `--force-with-lease` verifies the remote tag's current SHA matches what the local repo expects before clobbering. Plain `--force` would clobber regardless. The lease check catches concurrent overwrites by other agents (humans or sub-agents); without it, two concurrent release pushes could silently overwrite each other's work.

- **`delete + recreate + force-with-lease` pattern for stale remote tags** ‚Äî when a remote tag has a stale SHA from a prior pause-attempt, the recovery sequence is: (1) inspect the existing tag's content locally; (2) inspect the remote tag's SHA + commit history (`git ls-remote origin <tag>` + `git log <sha> -3`); (3) decide whether force-overwrite is justified (e.g., the stale tag is from a prior incomplete attempt); (4) locally delete + recreate with the production annotation; (5) `git push --force-with-lease origin <tag>`; (6) verify with `git ls-remote origin <tag>`.
