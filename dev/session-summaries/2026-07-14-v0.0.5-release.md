## Lessons Learned

(Captured in this session; candidate for `dev/LEARNINGS.md` codification pending Spencer's pick.)

- **LESSON-028 candidate — Field-specific verifier anchors** — Verifier regex anchors should target specific field names (e.g., `grep -E '^project\.version:'`) rather than broad substrings (e.g., `grep -E '^\s*version:'`). Broad substrings produce false positives from unrelated fields (e.g., `protocol.version` matching `^\s*version:` even though it's a different schema axis). The 2 false positives in this release (protocol.config.yaml 2nd version field + crates/skills test fixtures) are both instances of this pattern. **Resolution pattern:** when writing a verifier, use field-specific anchors; when investigating a verifier flag, always read the surrounding context (5-10 lines) to confirm the flag is a true positive before treating it as a code issue. Codification would prevent future agents from re-introducing the same broad anchors.

- **LESSON-029 candidate — `release.py` pre-flight is local-only** — `release.py` checks `git status --porcelain` for a clean working tree; the check is local-only (doesn't check the remote first). This is by design (script philosophy: "tag is local-first; push is a separate concern"; clean tree ensures the CHANGELOG auto-extraction is reliable for release notes). **Consequence:** when the script fails on a temp file or untracked file, the user must clean up the working tree before retrying. **Pattern:** run `git status --porcelain | wc -l` to confirm clean before invoking `release.py`; remove any untracked temp files (e.g., `dev/.tmp-*` or any other file used for the `git commit -F` / `git tag -F` file-based pattern) before running the script.

- **LESSON-030 candidate — `git commit -F` + `git tag -F` for complex messages** — When commit + tag messages contain special characters (backticks from the drift invariant phrase, em-dashes, etc.), the multi-`-m` pattern is brittle. The basher's shell escaping can mangle the backticks (observed in Stage 7 of this release). **Resolution:** write the messages to temp files via `write_file` (which handles all character escaping correctly), then use `git commit -F <file>` + `git tag -F <file>` (file-based, no escaping issues). The temp file approach also makes the message content easier to review (one less quoting concern). **Cleanup:** remember to `rm -f` the temp files BEFORE running `release.py` (per LESSON-029).

- **Prior lessons reinforced in this session:**
  - **LESSON-019 (release-only-versioning discipline):** the v0.0.5 release cut is a clean 7-file meta-file bump + CHANGELOG restructure, with no source-code changes. All version files mirror `package.json = 0.0.5` post-release. The discipline held: work-in-progress lives under `## [Unreleased]`; the bundle-identifier rename landed in v0.0.5 (per the forward-effective timing) not in v0.0.4 (per the pre-v0.0.4 deferral).
  - **LESSON-020 (Tauri version alignment to renderer):** the v0.0.5 release cut shipped `src-tauri/tauri.conf.json` at 0.0.5, aligned with the renderer `package.json` at 0.0.5. The Tauri app version and renderer version ship together as a single versioned artifact.
  - **LESSON-026 (str_replace multi-line phrase-split bug + byte-level anti-pattern):** the prior v0.0.4 session summary's LESSON-027 regression was caught and fixed via 6 str_replace edits, applying Option A (strip + abstract references) from the thinker analysis. This session's v0.0.5 summary was authored with the LESSON-026 lesson front-of-mind: all canonical references in the new file are on single lines, with abstract `<canonical anchor phrase>` placeholders used in place of verbatim citations.
  - **LESSON-027 (doc-drift substring-match invariant design):** the 5-anchor invariant was preserved through this release. The new v0.0.5 session summary has 0 anchors (verified post-write). The prior v0.0.4 session summary had a 5-anchor regression that was caught and fixed in Stage 1 of this session.
